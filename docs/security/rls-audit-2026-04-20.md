## Summary

The Archivist Method stores **paid content as markdown files on disk** (`the-archivist-method/`), served by Express with server-side tier checks against `quiz_users.access_tier`. There is **no** `archive_content`, `field_guide_content`, or `complete_archive_content` table — so "RLS on paid content" translates to:

1. RLS on `quiz_users` (the table that decides who sees what).
2. RLS on every table the **client** reaches with the anon key (`@supabase/supabase-js` in `src/lib/supabase.ts`).
3. RLS on every table holding PII (email, Stripe IDs, chat history, magic-link tokens) to block the anon role.

### Result: **5 critical gaps**. Production has at least one table (`quiz_users`) containing emails + unexpired magic-link tokens that is reachable via the public anon key, unless RLS was enabled in the Supabase dashboard outside this repo.

## Why `quiz_users` is the critical one

`src/lib/supabase.ts` exposes `VITE_SUPABASE_ANON_KEY` to every browser tab. Supabase exposes a PostgREST endpoint at `https://zxwswpopaonopntnjvlz.supabase.co/rest/v1/quiz_users` that any visitor can hit with that key. If `quiz_users` has **no RLS**, the default PostgREST grants for `anon` permit `SELECT`, which exposes:

- every buyer's email (GDPR-reportable incident)
- unexpired `magic_link_token` values → account takeover
- `access_tier` values → lets a scraper build a list of paying customers

I cannot verify the live DB state from the repo — **please confirm in the Supabase dashboard that `quiz_users` shows the "RLS enabled" lock icon**. If it does not, apply the proposed migration below before launch.

## Secondary concerns

- **`email_queue`** contains every subscriber's email address. Server cron only. Must not be readable by anon.
- **`purchases`** contains Stripe payment-intent IDs. Server only. Must not be readable by anon.
- **`portal_users`** contains `stripe_customer_id`. Server only.
- **`test_users`** is the god-mode allowlist. Server only.

All four share the same fix: enable RLS, write no policies — service role will still work because it bypasses RLS.

## Client-queried tables — audit notes

- `user_activity`, `activation_logs`, `brain_dumps`, `user_streaks` all use the own-row pattern (`auth.uid() = user_id`) which is correct **on the condition that the user has signed in via Supabase Auth** (so `auth.uid()` is populated). Tier-based gating (free vs paid) does **not** live on these rows — they are behavioural logs generated by any logged-in user. Tier gating for the *content* happens in `api/_content.ts` on the server.
- No anon-read or public-read policies were found on these tables.

## Proposed fix

Drafted as `supabase/migrations/20260420_enable_rls_on_unprotected_tables.sql`. **Do not apply until Aaron reviews.** The migration is idempotent; it only enables RLS and does not grant or remove any policies. Server code (which connects with the service role) is unaffected.

1. Open Supabase dashboard → Table Editor → inspect each table in the "RLS in repo?" column above; confirm "RLS enabled" lock icon.
2. Run in SQL editor:

   ```sql
   SELECT schemaname, tablename, rowsecurity
   ```

4. Then run, to confirm no stray public-read policies exist:

## Hardening follow-ups (not launch blockers)

- Rotate `SUPABASE_ANON_KEY` after RLS is confirmed so any key that leaked pre-launch becomes useless.
- Consider adding a tight SELECT policy on `quiz_users` keyed to `auth.email() = email` if any client-side read ever becomes necessary (currently none).
- Move `magic_link_token` to a separate `auth_tokens` table with service-role-only access, so it can never be exposed even by a future misconfig.
