## Architecture facts that drive policy design

1. **The server connects via `DATABASE_URL`** (Supabase connection pooler).
   The role used bypasses RLS (either `service_role` or a role with
   `BYPASSRLS`). All Express / Vercel serverless paths continue to work
   after RLS is enabled.
2. **The client does NOT use Supabase Auth.** `client/src/lib/supabase.ts`
   creates a client with the anon key, but `grep` confirms zero
   `supabase.auth.signIn/signUp` calls anywhere. Auth is handled by a
   custom JWT (`verifyAuthToken` in `server/portal/auth.ts`) validated
   server-side.
3. Because of (2), `auth.email()` returns NULL in every browser call
   today. "Own-row" policies written as `auth.email() = email` evaluate
   to FALSE from the browser, which is exactly what we want right now
   (block the anon key). The same policies activate correctly if
   Supabase Auth is wired up later — they're forward-compatible dormant.

## Per-table policy decisions (as approved in chat)

| Table | RLS? | Granted to anon / authenticated? | Policies |
|---|---|---|---|
| `quiz_users`   | ENABLE | `REVOKE ALL` | `SELECT own row` for `authenticated` via `auth.email() = email` |
| `portal_users` | ENABLE | `REVOKE ALL` | `SELECT own row` for `authenticated` via `auth.email() = email` |
| `email_queue`  | ENABLE | `REVOKE ALL` | none — service-role-only |
| `test_users`   | ENABLE | `REVOKE ALL` | none — service-role-only |

### Why `REVOKE ALL` in addition to `ENABLE ROW LEVEL SECURITY`?

Defense in depth. Supabase auto-grants `anon` and `authenticated` table
privileges (`SELECT` / `INSERT` / `UPDATE` / `DELETE`) when tables are
created through the dashboard. RLS alone blocks row access, but a
misconfigured future policy could re-expose data. Revoking the grants
means the policy is the *only* source of access — nothing leaks if a
policy is accidentally written too loose.

### Why no INSERT / UPDATE / DELETE policies on `quiz_users` / `portal_users`?

Because those mutations only happen server-side. If we added
`INSERT WITH CHECK (auth.email() = email)` we'd give an
anon-with-Supabase-Auth user the ability to create arbitrary rows —
not what we want. Keep mutations locked to the service role, full
stop.

## Vault tables (unchanged, already correct)

These already have RLS + own-row policies per
`src/sql/vault-schema.sql`. No change in this migration:

- `user_activity`
- `activation_logs`
- `brain_dumps`
- `user_streaks`

Caveat: their policies (`auth.uid() = user_id`) also rely on Supabase
Auth being populated. Since we don't use Supabase Auth, **the vault
tables' client inserts are effectively failing silently today.** Worth
flagging as a separate bug to investigate after the RLS lockdown ships.
